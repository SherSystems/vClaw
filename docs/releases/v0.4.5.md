## Security: dashboard isn't an open API anymore.

Two CRITICAL findings from the overnight security audit, both fixed in this release.

**The dashboard was bound to `0.0.0.0` with no auth.** `src/frontends/dashboard/server.ts` called `server.listen(port, cb)` with no host argument. On Tailscale-only deployments that was a soft mitigation â€” anyone on your tailnet could `POST /api/agent/command`, `/api/agent/approve`, `/api/chaos/execute`, `/api/migration/execute` without authentication. Fix: default-bind to `127.0.0.1`. Opt into network exposure with `RHODES_DASHBOARD_HOST=0.0.0.0` (or a specific interface). A loud warning logs when the binding is non-loopback.

**CORS allowed cross-origin POST / DELETE.** Any webpage the operator visited (a clickbait link, a phishing page, a compromised dependency's docs site) could fire a destructive RHODES action via CSRF â€” the dashboard echoed `Access-Control-Allow-Origin: *` on every method. Fix: CORS headers are now only emitted for `GET` and `OPTIONS`. Cross-origin mutations get blocked by the browser as the spec intends.

> *If you were relying on the dashboard being reachable from another machine on your tailnet, set `RHODES_DASHBOARD_HOST=0.0.0.0` in your `.env` and restart RHODES. The release won't surprise you with a 404; it'll surprise you with a "Connection refused" until you opt back in.*

**What this means for you:** the blast radius of a dashboard reachable on Tailscale is now operator-confined-by-default. Defense in depth still matters â€” see the HIGH findings below for the next layer.

## Planner: typed tools before SSH.

Last night's esxi-01 save failed at step 10 (`ssh_exec qm resume 200` with exit code 255 â€” no SSH key on the NUC to pranavlab). The Proxmox provider already had a typed `resume_vm` API tool, but the LLM picked `ssh_exec` instead because the typed tool's description said *"Resume a paused VM"* and didn't communicate the preference.

Fixed at three layers:

1. **System prompt** â€” a new hard rule #11 in `src/agent/prompts.ts`, sitting right after the AWS-instance-ID rule from v0.2.2:

   > **Proxmox VM lifecycle:** For Proxmox VM power operations (resume, start, stop, reboot, suspend, reset), ALWAYS use the dedicated Proxmox provider tools (`resume_vm`, `start_vm`, `stop_vm`, `reboot_vm`, `suspend_vm`, `reset_vm`). Do NOT use `ssh_exec` to run `qm` commands when a typed Proxmox tool exists for the same operation.

2. **Tool descriptions** â€” every Proxmox lifecycle tool now says *"Prefer this over `ssh_exec qm <verb>`"* inline. The LLM sees the steer in the tool list, not only in the system prompt. Anti-regression test asserts each of the six descriptions still contains `ssh_exec` â€” silently rewording one back to a stub trips the suite.

3. **Two missing tools added** â€” `suspend_vm` (safe_write) and `reset_vm` (risky_write). Proxmox doesn't expose reset for LXC, only QEMU; the client doc-comment notes this.

**What this means for you:** when RHODES wants to resume a paused VM, it uses the Proxmox API token RHODES already has, not an SSH path that may not be configured. Same fix applies to start/stop/reboot/suspend/reset.

## Telegram approval-pending deep links.

Every approval-pending alert RHODES sends through Supra â†’ Telegram now carries a deep link back to the dashboard with the plan id pre-targeted:

```
RHODES â€” approval needed
Action: qm delsnapshot 200 autosnap_2026-04-18 [risky_write]
â€¦
Approve at: http://100.73.129.96:7412/?plan=plan_01HXYZâ€¦
```

Tap the link on your phone, the dashboard opens, scrolls straight to the matching Pending Approvals card, pulses a Rhodes Blue ring around it for two seconds. If the plan was already approved or rejected by the time you tap (a coworker beat you to it; you came back from dinner), the dashboard shows a one-line toast â€” `Plan plan_01HXYZâ€¦ has been approved.` â€” instead of pointlessly scrolling to a missing card. The deep-link handler runs on initial page load **and** after every Pending Approvals re-render, so if you tap the link before the SSE has delivered the awaiting_approval entry, the highlight fires the moment the card appears.

Configure with `RHODES_DASHBOARD_URL` in `.env` (e.g. `http://100.73.129.96:7412` for the NUC over Tailscale). Empty (the default) suppresses the link and falls back to a plain `Plan: <id>` reference.

What's **not** in this release: real inline `Approve` / `Reject` buttons in the Telegram message itself â€” that requires a signed-callback flow through Supra's bot webhook. Deep link first; inline buttons next.

**What this means for you:** approve from your phone in three seconds. No more remembering the IP and hunting for the right card.

## Ops: NUC â†’ pranavlab SSH key bootstrap.

The 255 exit in last night's step 10 was the NUC having no SSH credential to reach the Proxmox host. Fixed operationally with three artifacts:

- **Key already generated on the NUC** (`~/.ssh/rhodes-pranavlab`, mode 600). Public key is in the runbook for paste-in-one-step installation.
- **`~/.config/rhodes/ssh-targets.json`** on the NUC defines the `pranavlab` target with `sudo_allowlist=[systemctl, journalctl, qm]` and tier overrides that let `qm <read-op>` execute as Tier 1 and `qm <power-op>` as Tier 3 (risky_write).
- **`scripts/test-ssh-to-pranavlab.ts`** â€” smoke test that exercises the SSH adapter end-to-end via `ssh_dry_run` then `ssh_exec pveversion`, with distinct exit codes for "no target registered" / "auth failed" / "command failed".

The one manual step left for the operator: install the public key in `root@pranavlab:~/.ssh/authorized_keys`. The runbook at `docs/runbooks/nuc-ssh-to-pranavlab-bootstrap.md` lists three install paths (sshpass-from-pranavserver, PVE web console SSH key flow, physical console paste). One-time bootstrap; once installed, the SSH adapter target is live after the next `systemctl --user restart rhodes`.

**What this means for you:** the next time the planner does pick `ssh_exec` (e.g. for a `qm` op that doesn't have a typed tool, or because the Proxmox API endpoint is degraded), the path is wired and the sudo allowlist is scoped to the verbs we actually need.

## Audit reports

Two comprehensive audits ran overnight against the v0.4.4 codebase. Reports live in `docs/audits/`.

### Security (`docs/audits/security-2026-05-14.md`)
- **2 CRITICAL fixed** in this release (the dashboard loopback + CORS issues above).
- **3 HIGH** documented, not fixed:
  - Dashboard has zero authentication on any endpoint â€” loopback narrows the blast radius but doesn't close it. A token / unix-socket auth flow is the right next step.
  - `SystemAdapter.configureService` shell-interpolates `service` and `config_path` directly into `systemctl ${service}` and `cat > '${configPath}'`, bypassing the SSH safety classifier (it calls `runProcess("ssh", ...)` directly).
  - `ChaosEngine.execute` checks `requires_approval && risk_score > 70` but only *updates a string* â€” execution proceeds regardless. The flagged scenarios are the high-blast-radius ones (`vm_kill`, `node_isolate`).
- **4 MEDIUM** documented (SSH governance binding, snapshot-name interpolation foot-gun, operator-controlled Supra URLs, plan_id reuse on `submitApiDecision`).
- **Confirmed clean inventory**: zero SQL injection (every query uses `better-sqlite3` parameterised bindings), zero hardcoded credentials in source, no identity-file logging, no path traversal in dashboard static serving, MCP server is stdio-only (not network-reachable), `.env` is read-only.
- **9 dependency CVEs** (6 moderate, 3 high) in transitive AWS-SDK / Hono / `path-to-regexp` / `ip-address`. None exploitable in RHODES today; `npm audit fix` is hygiene work.

### Correctness (`docs/audits/correctness-2026-05-14.md`)
- **0 CRITICAL** under the actual live state (`/api/agent/status` reports `dry_run: true, shadow_mode: true` at audit time).
- **2 HIGH** documented:
  - `src/governance/approval.ts` keys decisions by `plan_id` only â€” a prior plan-level approval auto-resolves any later per-step `requestApproval` for the same plan, defeating `policy.explicit_tiers`.
  - `src/agent/llm.ts` + `executor.ts` have no plan/step/LLM timeout for non-rollback-required actions. A hung LLM call wedges `agentCore.run` for up to the SDK default (10 min).
- **4 MEDIUM** documented:
  - SSE backpressure ignored (slow client â†’ starved eventBus).
  - service-http-probe restart-loop guard is in-memory only (lost on restart).
  - **Trigger collision** â€” `jellyfin-service-probe` and `vm_in_guest_diagnostic` both register `service_http_status / state_change / critical` and only one fires per anomaly. The v0.4.4 release-notes promise of auto-chaining was inaccurate; either matching should emit ALL playbooks or they should be sequenced via a chain field.
  - `setInterval` tick overlap.

Both audits explicitly state the codebase is safe to keep running overnight under the current (shadow-on) live state. **Fixes for the four HIGH findings are queued for v0.4.6** before the next shadow-off run.
