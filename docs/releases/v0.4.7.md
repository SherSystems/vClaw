# RHODES v0.4.7 — Login.

Released **2026-05-14**.

v0.4.5 narrowed the dashboard's blast radius (bind-loopback-by-default). v0.4.6 fixed every other HIGH finding from the audits. v0.4.7 closes the last one: the dashboard now requires authentication. Two-tier roles. Admin badge in the header when you're signed in as admin. Bootstrap flow on first run.

This is the release that lets you put RHODES on something that isn't tailnet-only and not feel queasy.

---

## What you sign in to.

Open the dashboard for the first time after upgrade and you get a setup page — the bootstrap flow. Pick a username and password (8+ chars). Submit. That account becomes admin. The cookie sets, the app mounts, you're in.

Subsequent visits: a calm centered login card with the RHODES lockup at top. Username, password, "Sign in." Get it wrong five times in 15 minutes and the rate-limiter returns 429.

When you're signed in as admin, a small mono **ADMIN** pill in Rhodes Blue surfaces in the header next to your username. Viewers don't get the pill. That's the indicator.

## Two roles.

- **admin** — can do everything. `POST /api/agent/command`, `POST /api/agent/approve`, `POST /api/chaos/execute`, the topology mutations, user management. Every mutating verb on every non-public path is admin-only.
- **viewer** — can read everything. Incidents, providers, playbooks, audit log, pending approvals (visible but the Approve/Reject buttons return 403 if clicked). Every GET on every non-public path requires at least viewer.

Two endpoints are public by design: `/api/healthz` for external monitors, the SPA shell at `/` (the client decides whether to render the login page or the app once it sees the auth state). The brand assets and SPA bundle (`/brand/*`, `/assets/*`) are also public so unauthenticated clients can paint the login page.

If you ever need the old wide-open behavior — for debugging, recovery from a locked-out admin account, or a one-off migration — set `RHODES_AUTH_DISABLED=true` in `~/rhodes/.env` and restart. The startup log shouts about it.

## Adding more users.

CLI for now:

```bash
ssh pranav@100.73.129.96 'cd ~/rhodes && \
  RHODES_BOOTSTRAP_TOKEN=<your-token> npx tsx scripts/rhodes-user-add.ts alice viewer'
```

The `RHODES_BOOTSTRAP_TOKEN` env var is auth for the CLI — required once at least one user exists. You can also generate one with the `--token` flag.

A `POST /api/auth/users` admin-authenticated endpoint is queued for v0.4.8 so you can manage users from the dashboard. For now the CLI is the path.

## How it's wired under the hood.

- **Sessions** — HS256 JWT in an httpOnly + SameSite=Strict cookie (`rhodes_session`), 24h expiry. Secret from `RHODES_SESSION_SECRET` env, or auto-generated to `~/.rhodes/session-secret` (mode 0600) on first boot if unset.
- **Passwords** — `bcryptjs` at cost 12. Pure JS, no native build deps. Hashes stored in `~/.rhodes/users.json` (mode 0600 enforced; refuse to read if mode is broader).
- **Rate-limit** — 5 login attempts per IP per 15 min, sliding window in-memory.
- **CORS** is still scoped to GET/OPTIONS from v0.4.5; cross-origin POSTs still get blocked at the browser. The auth layer is the additional bumper on top.

## Production note

If you deploy to a host that gets re-created (a VM template, a container that doesn't persist `~/.rhodes/`), **set `RHODES_SESSION_SECRET` to a stable value in `~/rhodes/.env` BEFORE deploying**. Otherwise every re-create rolls a new secret and invalidates all existing sessions. Homelab single-instance? Auto-generated is fine.

## Deploy

```bash
ssh pranav@100.73.129.96 '
cd ~/rhodes
git pull && npm ci && npm run build
npm --prefix dashboard-v2 ci && npm --prefix dashboard-v2 run build
systemctl --user restart rhodes
'
```

**Then immediately open the dashboard at http://100.73.129.96:7412/** — you'll see the bootstrap page. Set your admin credentials there. After that, all mutating endpoints (Approve, Remediate, anything in Chaos) require you to be signed in.
