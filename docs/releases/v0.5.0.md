# v0.5.0 — Slack as the customer-facing SRE surface

**Released:** 2026-05-15
**Branch:** `feature/slack-integration`
**Built on:** v0.4.7

This release makes Slack the operator's primary control surface for incident
response. A VM goes down, RHODES posts a Block Kit card in the operator's
DM, autopilot or a playbook recovers the VM, and an LLM-written postmortem
lands as a threaded reply — all without a human in the loop. The operator
can comment back in the thread and the comment shows up on the dashboard
ticket attributed to their real name.

## What's new

### Slack as the operator surface

- **`SlackProvider`** — outbound Block Kit messages with per-kind channel
  routing, severity badges, structured label fields, and "View ticket"
  deep-links. Configured via `RHODES_SLACK_BOT_TOKEN` +
  `RHODES_SLACK_DEFAULT_CHANNEL` + `RHODES_SLACK_CHANNEL_BY_KIND` (JSON map
  of alert kind → channel id, for routing `ticket_opened` to an operator
  DM channel while `approval_needed` lands in a team channel).
- **Multi-provider Notifier** — primary heartbeat provider (`supra`,
  `telegram_direct`, or `slack`) plus a Slack secondary for the team
  surface. Set `RHODES_ALERT_PROVIDER=slack` if Slack is your only
  surface; the secondary-slack guard avoids double-fanning.
- **Conversational thread replies** — `@rhodes investigate vmid N` in a
  channel triggers the LLM agent and posts one consolidated reply in the
  thread (not per-step spam). DM thread replies under a ticket card sync
  back as ticket comments AND also reach the agent for follow-up
  questions.

### Tickets — engineering-grade incident logbook

- **`RHODES-YYYY-NNN` ids** with an atomic year-scoped sequence
  allocator (no gaps under concurrency).
- **Auto-generated postmortems** — when an incident resolves, the
  postmortem hook calls the LLM and stores a 3–6 sentence
  technical/calm-voiced paragraph on the ticket. Falls back to a
  "regenerate from dashboard" note on timeout, never blocking the
  resolve.
- **Resolved-DM as a thread reply** — the postmortem lands as a Slack
  thread reply under the original ticket-opened DM, with a green
  ✅ RESOLVED header, the postmortem in the section body, a context
  line describing the state transition, and a "View ticket" deep-link.
- **Bidirectional thread sync** — operator replies in the DM thread
  (or `@rhodes` in a channel thread) land as comments on the ticket.
  Authors render as the resolved Slack display name (e.g. "Pranav")
  via cached `users.info` lookup — not the raw user_id.
- **Dashboard comments** are attributed to the authenticated session
  username, not a generic "operator" label.
- **Dashboard `Tickets` tab** — top-level page in the sidebar under
  the Operations group, with filters for All / Open / Healing /
  Resolved / Closed.

### Incident lifecycle — autonomous from detect through close

- **VM-state recovery** — incidents transitioning to `runtime_status:
  "running"` resolve automatically. Equally handles boot-eval
  `discovered_state_change` and live `running → stopped` transitions
  (which used to emit `anomaly_type: "threshold"` and fall through
  the recovery gate).
- **Failed-incident rescue** — when the autopilot rule restarts a VM
  while the healing engine's playbook fails on a parallel path, the
  rescue loop resolves the incident if the VM is healthy, fires the
  postmortem-on-resolve hook, and DMs the operator. Architectural
  fix to keep both systems from leaving incidents stuck.
- **Unified `IncidentCoordinator.resolveIncident()`** — every
  resolve path (healing-engine fast-path success, healing-engine
  standard playbook success, recovery-loop state recovery,
  recovery-loop numeric threshold) goes through one bundled method
  that calls `incidentManager.resolve()` AND fires the
  ticket-resolve hook. No more cases where a resolve quietly skips
  ticket sync.
- **Per-VM sample coalescing** — `detectVmStateChanges`,
  `evaluateInitialState`, and the rescue path now pick the freshest
  sample per (vmid, node), avoiding the stale-series flap where a
  `runtime_status="stopped"` series and `runtime_status="running"`
  series would oscillate every poll and spam tickets.

### Operational

- **Dashboard auth** survives service restarts (JWT signing secret
  in `RHODES_SESSION_SECRET` env or a persisted `~/.rhodes/session-secret`
  with mode 0600 enforcement).
- **Slack-relay shim** at `shim/` — signature-verifying edge service
  that fronts the Slack events/interactivity/command endpoints. Run
  it on the operator's tailnet behind a Cloudflare tunnel and Slack
  webhooks never reach internal infra directly.
- **Postmortem regeneration endpoint** — `POST /api/tickets/:id/regenerate-postmortem`
  re-runs the LLM if the operator wants a fresh take.
- **AdminBadge + UserMenu** in the dashboard header — auth status
  visible, signout one-click.

## Architectural rules going forward

1. Never call `incidentCoordinator.incidentManager.resolve()`
   directly. Always go through `incidentCoordinator.resolveIncident()`.
2. Postmortems are LLM-generated on resolve; operators can edit
   before close. The close endpoint requires a non-empty postmortem.
3. Slack is the customer-facing surface. Personal heartbeat surfaces
   (Telegram/Supra) get the full event stream; the Slack team
   channel sees the team-channel allowlist only (no
   per-step execution spam).
4. VM restart is autonomous — no approval gate. Approval gates are
   reserved for destructive/cross-system actions.

## Known limitations carried into v0.5.1

- RCA hallucination on operator-initiated shutdowns (no
  maintenance-mode signal — every operator stop reads as a crash).
- `incident_failed` Block Kit cards in the team channel render
  bare incident UUIDs instead of `RHODES-YYYY-NNN` ticket ids.
- Dashboard-v2 vitest setup can't resolve `react` across the
  monorepo boundary; 4 file-level test failures in `dashboard-v2`
  test files are pre-existing, not regressions.
- Ephemeral `*.trycloudflare.com` tunnel dies on cloudflared
  restart — the Slack manifest URLs are brittle to this. A named
  tunnel + custom domain is queued.
- The current schema doesn't auto-close tickets when an incident
  reaches terminal `failed` status; the rescue loop handles the
  common case where the VM has actually recovered.
