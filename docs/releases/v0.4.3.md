# RHODES v0.4.3 — Safer storage. Smarter SSH. Eyes inside the VM.

**Released:** 2026-05-13
**Tagline:** Infrastructure, executed — now with a rollback.

This release hardens the recovery paths that operators rely on most.
Snapshot pruning ships with a built-in safety net. SSH sessions
sudo-escalate only when you've explicitly opted in, and only at the
tier you already approved. And RHODES now diagnoses an unreachable
service from inside the VM, the way an SRE would.

---

## Snapshots, safer.

When RHODES recovers a thin-pool-paused VM, it now takes its own
safety snapshot before any prune, holds the previous safety snap
across the run as a rollback target, and refuses to prune below a
one-snapshot retention floor. The recovery path that worked on
2026-05-12 is now the recovery path you can't accidentally weaponize.

The retention floor is enforced in `filterDeletableCandidates()` —
the newest non-`current` snapshot is excluded from candidates, so a
VM with one remaining snapshot produces an empty plan instead of a
risky one. The pre-remediation `rhodes-safety-<ISO>` snapshot is taken
as `safe_write` Tier 2; the prior safety snap only deletes after a
successful resume + verify; on a failed resume, every safety snap is
preserved as a rollback target.

**What this means for you:** every storage-pause remediation now
ships with a rollback button built in — no operator discipline

## RHODES can now act like a human SRE inside a VM.

A new in-VM diagnostic playbook runs the nine commands a human
operator runs every time a service stops responding — in parallel,
through a single SSH session — and reduces the outputs to a typed
classification: out of disk, out of memory, OOM-killed, service
crashed, service not listening, kernel error, I/O error,
boot-looping. The plan it produces is tier-classified, approval-gated,
and refuses to auto-act on boot loops or kernel errors.

The decision module lives at `src/playbooks/vm-diagnostic.ts` —
operators can drive it programmatically today. v0.4.4 wires it into
the healing registry so an HTTP-probe failure auto-routes here.

**What this means for you:** "service down, host alive" goes from a
30-minute diagnostic session to a one-click approval.

See: [docs/playbooks/vm-diagnostic.md](../playbooks/vm-diagnostic.md).

## Sudo, when you've asked for it.

The SSH adapter now tries every command unprivileged first. When the
target's user lacks privilege for a specific verb, RHODES retries
with `sudo -n` — but only if the verb is in that target's
`sudo_allowlist`, only if the failure looks like a privilege error,
and only if the sudo'd form classifies at the *same* governance tier
as the unprivileged form. Any escalation that would jump tier is
refused outright — the caller's approval was for the lower tier and
RHODES will not silently promote it.

No more permanent root SSH sessions. No more "I can't diagnose this
because the agent's user can't read `/var/log/syslog`". A per-verb
NOPASSWD line in your sudoers, declared once in the target's config,
unlocks exactly the verbs you authorized — and the audit trail proves
which escalations fired.

**What this means for you:** least-privilege SSH that still gets the

## Per-target governance.

Each SSH target may now carry `tier_overrides` — a per-host floor
tier plus a per-tag/per-command override map. Bump everything on a
production host up to `risky_write` even for an `ls`. Pin a specific
verb on a sandbox down to `safe_write`. The audit trail records both
the base classifier verdict and the override key that fired, so an
auditor can answer "why was this command treated as Tier N on host X"
in one lookup.

Overrides can raise OR lower a tier — except `never`, which is
permanently `never`. That one is non-negotiable.

**What this means for you:** the safety model adapts to your fleet's
risk profile, host by host.

See: [docs/ssh-governance.md](../ssh-governance.md).
