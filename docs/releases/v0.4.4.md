# RHODES v0.4.4 — Approve. Remediate. Recover.

Released **2026-05-13**.

If v0.4.3 was the brain — three new playbooks, snapshot retention, governed sudo — v0.4.4 is the hands. Detection without action is half a product. This release closes the loop you watch.

---

## The Approve button you can actually click.

The agent has known how to ask for approval since v0.4.2. The dashboard didn't know how to render the question. Until now plans queued silently at `/api/agent/pending-approvals` and the only way to see them was `curl`. That gap is closed.

A new **Pending Approvals** panel sits above the tab bar — visible from any view, hidden when empty. Each pending plan is a card: the action in JetBrains Mono, a color-coded tier badge, the agent's reasoning, a key:value params grid, and two buttons. **Approve** ships the plan. **Reject** kills it. Both update live via SSE; refresh the page and the panel rehydrates from the catch-up poll.

Plans that include a `rhodes-safety-*` snapshot step (the v0.4.3 retention rule) render with a 🛡️ "SAFETY SNAPSHOT" badge and a green border instead of teal — visible proof that RHODES is bringing more discipline to the recovery than you would at 3am.

> *Screenshot placeholder: pending-approvals card for `qm delsnapshot 200 autosnap_2026-04-18` with the prepended safety-snap shown shielded above it.*

**What this means for you:** when RHODES wants to do something that requires your sign-off, it shows up on the screen you're already looking at. One click executes. Same approval flow whether the plan came from a playbook auto-firing or from the new Remediate button.

## Remediate from anywhere.

Every open incident card now has a **Remediate** button. Click it and the dashboard builds a natural-language prompt from the incident's labels — `vmid`, `node`, `reason`, `service_name` — and posts it to `/api/agent/command`. The agent plans the recovery, the plan lands in Pending Approvals, you approve.

`vm_status: paused_io_error` becomes *"Investigate VM ninja-bot (vmid 210) on pranavlab — currently in paused (io-error). Run the proxmox-storage-pause playbook to diagnose snapshot bloat, propose a remediation plan, and wait for approval before executing."*

`service_http_status` becomes *"Investigate service jellyfin — HTTP probe failing. Run the in-VM diagnostic playbook against the target VM. Propose remediation and wait for approval."*

You don't have to author the goal. The dashboard knows the shape from what RHODES already detected.

> *Screenshot placeholder: an incident card mid-click — Remediate button transitioning to "Planning…" disabled state.*

**What this means for you:** detection and action are now one click apart. The "I see the incident, where's the button" question doesn't get asked anymore.

## Self-healing when things heal themselves.

`resolveRecoveredIncidents` had a real bug: incidents synthesized by the boot-eval pass (the Agent G fix in v0.4.2) never auto-closed when the VM returned to running. The original recovery logic was a numeric threshold — `latest.value < trigger_value * 0.7` — which is meaningless when the metric is a state-change marker (1 ≡ "transitioned").

v0.4.4 adds a state-change recovery path. When the latest `vm_status` sample for the same `vmid+node` shows `runtime_status ∈ {running, ok}`, RHODES resolves the incident with reason `"VM <name> state recovered: paused_io_error → running"` and fires `AlertResolved`. The numeric-threshold path for non-state-change metrics (memory, CPU, etc.) is untouched.

Live test: as of this morning the production NUC had two stale `paused_io_error` incidents — `ninja-bot` (vmid 210) had been showing the warning triangle for 24 hours after the VM had actually come back. After this fix deployed, the next health poll closed it automatically.

**What this means for you:** the dashboard stops lying about state. Triangles clear when the VM is fine.

## Probes wired end-to-end.

The new in-VM diagnostic playbook (`src/playbooks/vm-diagnostic.ts`, v0.4.3) is now registered as an auto-firing rule in `src/healing/playbooks.ts`. Trigger: `service_http_status` state_change critical, `requires_approval: true`, `cooldown_minutes: 15`. Registry count goes from 8 to 9.

The chain RHODES already had:

```
HTTP probe fails 3x → service_http_status state_change → jellyfin-service-probe
  ↓
  jellyfin-service-probe restarts the service
  ↓
  still failing? → escalate
```

The chain it has now:

```
HTTP probe fails 3x → service_http_status state_change → jellyfin-service-probe
  ↓
  service restart fails OR not the issue → vm_in_guest_diagnostic auto-fires
  ↓
  SSH in, 9-command sweep, classify failure mode, propose plan
  ↓
  operator approves in Pending Approvals → execute → re-probe
```

The two playbooks share the same trigger so they fire together — the http-probe handles the simple "just restart it" case, the diagnostic playbook handles everything underneath (disk full, OOM, kernel error, boot loop, service not listening).

**What this means for you:** when Jellyfin (or any HTTP service RHODES watches) goes sideways and a restart doesn't fix it, RHODES does the same investigation a human SRE would — looks at logs, disk, memory, dmesg, listening ports — and surfaces a plan instead of just paging you.
