## Thesis

RHODES is the **AI brain for infrastructure** ‚Äî a substrate-agnostic autonomous agent that operates across vSphere, AWS, Azure, K8s, and Proxmox via an infrastructure ontology (graph) that captures the customer's entire environment, queryable by the agent before it acts. The headline value to a buyer is that it does the labor of a senior infrastructure engineer (cluster upgrades, healings, evacuations, coordinated multi-substrate rollouts) at material discount to the VMware-consultancy alternative.

v0.5.0 made RHODES a smart notifier + auto-restarter (one VM at a time). **v0.6 builds the graph as the substrate the orchestrator queries, and the operational primitives that the v0.7 cluster upgrade orchestrator will compose.** Without the graph, the orchestrator is substrate-aware-but-context-blind. With it, the orchestrator becomes the autonomous senior engineer.

## Architecture decisions (locked)

### 1. The brain: infrastructure ontology / graph

Continuously-updated graph of every Resource + Relationship across all substrates. Resources have universal typed columns (id, provider, type, name, state, timestamps) + per-type JSON `properties` for substrate-specific fields. The agent queries the graph *before* generating plans.

**Schema decisions:**

- **Cross-provider identity via `manifests_as` edges** (JupiterOne's `IS` pattern). Each provider gets its own Resource; an explicit typed edge connects perspectives of the same logical thing. Shared interface labels (`:ComputeInstance`, Cartography pattern) for cross-provider queries that don't need traversal.
- **Universal fields + JSON properties blob**, with two **mandatory guardrails**: (a) per-type JSON schema registry validated at write time (mirrors `aws-config-resource-schema`); (b) promotion protocol ‚Äî fields in WHERE clauses >monthly get promoted to Postgres generated columns, then to typed columns via expand-and-contract. Skip per-type extension tables entirely. Cap blob size at 4‚Äì8 KB.
- **Per-type closed state enums** (`{running, stopped, error, unreachable, unknown}`-style) with mandatory `unknown` member, append-only versioning. Pair every state with `state_observed_at`. Add optional `conditions[]` sidecar (K8s shape) for orthogonal signals one enum slot can't carry (`Reachable`, `AdapterHealthy`, `InSync`).
- **Per-type freshness windows + 3 query modes** (`fresh` / `stale-allowed` / `historical`). Decouple "freshness window" (consumer contract) from "polling interval" (collector impl). Expose `last_observed_at` per record, not just `is_stale` bit. Fresh-mode escape hatch: synchronous live fetch when not present. Periodic full reconciliation distinct from event stream. Default full-resync ~1h (not the controller-runtime 10h footgun).

### 2. The hands: substrate-agnostic primitives

The orchestrator never imports a provider-specific client. Adapters implement a small typed contract:

```
evacuate_workload(target, provider, mode)    // mode = 'live_migrate' | 'evict' | 'replace'
enter_maintenance(host, provider)
exit_maintenance(host, provider)
remediate(host, provider)
rollback(plan, step)
```

**Capability discovery as first-class resource.** Each adapter publishes what it can/can't do; the orchestrator's planner reads capabilities and plans around constraints. **Do not pretend semantically-different primitives are identical** (the AWS Security Groups stateful-per-NIC vs Azure NSGs stateless-per-subnet lesson).

**Wrap, don't replace, native upgrade engines.** vLCM remediate, EKS UpdateNodegroupVersion, AKS upgrade already encode years of edge cases. Adapters drive them; they do not reimplement them.

### 3. The substrate: durable execution from day one

**Adopt Temporal (or river / inngest / cloudflare workflows) as the orchestrator runtime ‚Äî added to v0.6 scope.** Netflix Spinnaker's transient deploy failure went 4% ‚Üí 0.0001% by adopting durable execution. Roll-your-own state machines bite at p95. This is the single biggest architectural delta from the prior v0.6 scoping draft.

### 4. The senses: per-substrate event-source attribution

Per-substrate event adapters tag every state change with who/what triggered it:
- vCenter event API (VMware)
- CloudTrail (AWS) ‚Äî `userIdentity` field
- Azure Activity Log ‚Äî `caller` field
- K8s audit log
- Proxmox task log

Common attribution module on top. RHODES only opens incidents on unattributed or unexpected changes. The slash-command silence path is the escape hatch, not the primary mechanism. **Solves the v0.5.1 RCA-hallucination bug AND is a hard requirement for v0.6.5+ upgrade orchestration (otherwise RHODES self-alarms during every host maintenance).**

### 5. The voice: Slack-as-SRE-surface (continues from v0.5.0)

Block-Kit approvals, ticket DM threads, conversational @-mentions, comment-sync. No changes needed for v0.6.

## Cross-cutting concerns

### Action tiering for upgrades

5-tier governance (read/safe_write/risky_write/destructive/never) works for most things but is undersized for plan-scope sequences. **Add `plan_approval` governance hook**: fires once at plan creation, authorizes a sequence. Per-step governance still runs but uses plan_approval as a credential ‚Äî "this step is part of approved plan X." Audit trail records both.

### Observability of long-running plans

Extend the existing `Active Plan` tab (`dashboard-v2/src/components/ActivePlan.tsx`) to render upgrade state-machine progress. Keeps the dashboard simpler than adding a separate Upgrades tab.

### Backwards compatibility / customer migration

vCenter floor: 7.0u3+ only (image-based LCM lives there; earlier versions used baseline-based vUM which is deprecated). **If customer is below floor, RHODES auto-upgrades vCenter as a prerequisite step** (not a "sorry, do this yourself first" message).

### Cloud sandbox

LocalStack (AWS) + Azurite (Azure) + KinD (K8s) for CI write-path tests. The $4.50/mo canary stays read-only for production behavior validation. Customer credentials when there's a real customer.

## Out of scope for v0.6

- NSX-T / NSX-V upgrades ‚Äî separate orchestrator, separate cycle
- vSAN-specific health gates beyond what LCM ships with
- Aria Operations / vROps integration ‚Äî LCM provides enough signal
- Windows guest patching / linux distro upgrades inside VMs ‚Äî separate problem space
- Migration of workloads between hypervisors ‚Äî already covered by existing migration subsystem

## What's NOT changing

- Slack DM thread surface from v0.5.0 stays the operator's main control point
- Five-tier governance model is the base; extend with `plan_approval`, don't replace
- Provider adapter pattern stays ‚Äî every new substrate capability goes through `src/providers/<substrate>/`, never around it
- v0.5.0 invariants (autonomous restart with no approval, postmortem on resolve, conversational @-mentions) keep working as-is
